# 自动化分析可行性评估报告

## 1. 爬虫出口审计 (Crawler Exit Logic)
- **退出钩子位置：** 爬虫任务最终在 `views.py` 的 `_run_spider_background` 函数中完成。
- **返回数据结构：** `run_spider_service` 执行完毕后会返回两个具体的 CSV 文件存放路径（`video_path` 和 `comment_path`）。
- **任务结束代码行：**
  - 第 `1542` 行调用了入库核心逻辑，执行了：`import_result = import_data_service(theme_name, video_path, comment_path)`。
  - **建议的钩子插入点：** 在第 `1544` 行的 `if import_result['success']:` 分支内。例如当判定成功且控制台输出成功入库后，即可插入触发“增量情感分析”的钩子代码。

## 2. 分析服务审计 (Sentiment Analysis Service)
- **底层打分函数：** `views.py` 中定义的 `calculate_refined_sentiment(text)` 和 `extract_semantic_features(text)`。
- **输入参数：** 目前已经存在用于处理情感分重算的 `recalculate_sentiment_api` 函数，但它接收的仅仅是 `theme_label` 并对整个主题执行全表的重新计算。
- **增量处理能力：** 当前的 `recalculate_sentiment_api` 函数本身**不具备**“增量处理”能力，它查询条件是 `Comment.objects.filter(theme_label=theme_label)`，并没有过滤 `sentiment_score == 0.5` 的逻辑，这将导致大量的覆盖写和重复运算。

## 3. 状态与进度同步审计 (Status & Progress Sync)
- **共享缓存问题：** 爬虫抓取与 AI 分析并**没有共享**同一个缓存键。
  - 爬虫使用缓存键 `spider_progress`（包含 `percent`、`status`）。
  - AI 使用后台定时轮询通过数据库查询状态，并在前端接口 `get_analysis_status_api` 中汇聚（如 `analysis_status=0/1/2`）来判断最终进度与全局状态。
- **状态转换逻辑：**
  - **0：** 未处理（由爬虫的批量保存函数 `UnifiedPersistenceManager.save_video_record` 初始化写入）。
  - **1：** 处理中（由守护线程 `AIAnalysisWorker` 在处理视频多模态特征前取得任务标记）。
  - **2：** 已完成（由守护线程处理完毕后保存状态）。
  - **-1：** 失败（在分析超时或异常时保存状态）。

## 4. 并发与线程审计 (Concurrency & Threading)
- **死锁或显存溢出风险：** 
  - 当前架构中，由于采用了独立的后台守护线程 `AIAnalysisWorker`，异步的 AI 模型特征处理并没有和爬虫所在的 `_run_spider_background` 线程产生冲突，直接的死锁可能性较低。
  - 但是，如果选择在爬虫结束回调处**串行直接执行**大量评论的情感评分（SnowNLP），极大概率会导致内存溢出和爬虫线程被长耗时运算死锁。
- **批量拆分建议：** 
  - SnowNLP 本身纯依赖 CPU，大量数据运算耗时较长。
  - 强烈建议**必须分批处理**（例如采用类似 `save_comment_batch` 每次取出 200 条数据）或新建一个常驻的 `SentimentWorker` 来异步消费 `sentiment_score=0.5` 的增量数据。

## 5. 当前版本潜在 Bug（0.5 默认值无法被覆盖）
1. **重算API逻辑漏洞：** `recalculate_sentiment_api` 中目前存在全量遍历瓶颈。如果不加以改进为增量处理，随着主题评论增加，会引发 OOM 或直接请求超时。
2. **幂等性导致“沉默的失败”：** 在 `services/data_manager.py` 的评论入库模块中使用了 `ignore_conflicts=True`（批量插入时忽略冲突）。这意味着那些已经有了实际情感打分的旧评论不会被新抓取并默认赋值为 `0.5` 的操作覆盖，这就要求后面的“自动触发钩子”必须要显式且精确地 `filter(sentiment_score=0.5)`，否则会因为主键冲突机制导致真正的计算永远无法执行。
3. **浮点精度危险：** 直接做 `sentiment_score == 0.5` 的等值过滤在某些 Django/SQLite 存储结构中可能存在隐患，更好是新增 `is_analyzed=False` 标志位或判断范围（`0.49 < x < 0.51`）。

---
**审计结论：**
整体“抓取后自动触发”的基础环境与模型均已完善可用。**建议的改造路线**为：在 `_run_spider_background` 的成功钩子 `if import_result['success']:` 后激活或插入专用的异步任务（如 Celery 或 `threading.Thread`），专门过滤查询条件为 `sentiment_score=0.5` 的那部分记录执行批量（Batch）的 `calculate_refined_sentiment` 操作。
