## 🎯 1. 特征工程溯源 (Feature Engineering Discovery)
**审查目标：** 定位底层维度映射机制与动态特征拼接逻辑

### 1.1 独热编码(One-Hot)与物理特征处理
1. **源码位置：** `d:\renhangxi_tiktok_bysj\services\predict_service.py` —— 预测服务入口 (`predict_digg_count` 方法)。
2. **确认映射机制：**
   - 不依赖独立的 `Dictionary` 文件，也**不依赖于数据库 `Theme` 表的外部映射**。目前在内存中所采用的映射机制完全**硬编码依附**于加载在内存中的增量特征名 (`feature_names_in_`)。
   - `predict_service.py` 的第 124-128 行，程序从离线保存下的 `model.feature_names_in_` 提取出所有以 `theme_` 开头的特征名作为 `known_theme_cols`。
   - 第 240-245 行基于前段提取出的特征名字段(`ordered_feature_names`)做循环。通过字符串剥离（`column_suffix = col.replace('theme_', '')`）与当前 `theme_label` 直接精确比对来确立 One-Hot 位点（`X_final.at[0, col] = 1.0`）。

### 1.2 特征拼接次序 (Concatenation / DataFrame Creation)
1. **源码位置：** `predict_service.py` 阶段 C，第 213 行 (`X_final = pd.DataFrame(0.0, index=[0], columns=ordered_feature_names)`)
2. **拼接特征序列：** 未直接采用 `numpy.concatenate`。实际过程采用 `Pandas` 的宽表分配模式，以数学描述方式表达，**拼接次序 ($X_{final}$) 完全由** `ordered_feature_names` 即**模型训练时的输入拓扑**硬性决定（通过 `get_dummies` 在训练脚本中生成）：
   - $X_{base}$（基础物理维度，如：`follower_count`, `publish_hour`）
   - $X_{derived}$（人工聚合衍生，如：`visual_impact`, `sensory_pace`, `sentiment_intensity` 等）
   - $X_{theme}$（动态扩展的One-Hot维度表）
   - $\sum X_{final} = [X_{base}, X_{derived}, X_{theme}]$，按特征提取器 (`scaler` / `feature_names_in_`) 中的排列强制归位。

## 🎯 2. 模型加载与资源管理 (Model Lifecycle)
**审查目标：** 评估内存管理及负载承受能力

### 2.1 确认单例模式与模型加载
1. **源码位置：** `d:\renhangxi_tiktok_bysj\services\predict_service.py` (第 39-87 行)。
2. **确认加载机制：** 是**真正的单例模式 (Double-Checked Locking, DCL)**。
   - 包含着类级属性 `_instance`、互斥锁 `threading.Lock()` 与对象状态锁 `_initialized`。这代表模型**不会在 `apps.py` 启动或每个 API 命中时加载**，而是在整个 Django 进程存活期首次对 `DiggPredictionService` 请求实例化时通过 `joblib.load(model_path)` 临时进行且仅发生1次，具备良好的内存规避特性及防竞争击穿性能。
3. **关键发现：**
   - 依据清单文件 `artifacts\version_manifest.json` 可见，虽然核心加载器设计存在 `XGBoost` 兼容代码，但真实调用的算力核心当前为 `LightGBM`，具有 29 个降频特征位，该特征与当前 API 数据可能存在脱节。

### 2.2 确认显存/内存策略
1. **源码位置：** `d:\renhangxi_tiktok_bysj\renhangxi_tiktok_bysj\douyin_hangxi\utils\video_analyzer.py`
2. **确认资源回收逻辑：** **有严格且显式的显存回收机制。**
   - 利用 `torch.cuda` 并发进行基础帧亮度极差渲染和 ASR (Whisper) 时，使用了资源回收的兜底保护。系统在 3 个位置（178 行捕捉 OOM异常、193 行分析完单视频视频截断的 `finally` 块、307 行 Whisper 生成后）严格触发了 `torch.cuda.empty_cache()` 用于强制回收集游离在 VRAM 内的垃圾堆缓存，有效保全了 Django Web 会话稳定。

## 🎯 3. 事前决策逻辑审计 (Pre-publishing Logic)
**审查目标：** 发布时间前瞻性与主题分布建议逻辑

### 3.1 确认基准水位计算机制
1. **源码位置：** `d:\renhangxi_tiktok_bysj\ml_pipeline\theme_baseline_engine.py` (第 98-106 行) 与 `views.py` `predict_api` 视图 (第 1133-1147 行)。
2. **逻辑机制：** 它是**实时聚合所有爬虫数据**的。
   - 当 API 请求触发时，系统会经由 `Video.objects.filter(theme_label=theme_name).values('digg_count')` 动态计算特定主题的全部数据水位。
   - 通过调用 `ThemeBaselineCalculator` 计算统计学参数，并未引入离线缓存。在获得当前赛道的局域计算后取得分布：$Y_{local} = \text{Percentiles}(p25, p50, p75)$。

### 3.2 确认时间决策算法发现
1. **源码位置：** 系统各文件关于 `publish_hour` 与 `create_time`。
2. **逻辑机制：** **无深度分桶 (Binning) 或概率密度 (PDF) 推荐算法结构。**
   - 对于输入（`predict_api` 传入的 `publish_hour`）：该时间仅仅是从用户传入，被平坦化处理为了模型可以直接吃掉的**单一连续标量或粗分类**放进特征中。
   - 系统未对 `create_time` 在本地统计层面展开过“几点更适合发布本视频”的时频极轴分析或核密度估计算法。时间发布决策算法完全**被外包化推测**：它将 `p_data` 的全量矩阵抛给了 LLM 的 Prompt (`utils\llm_service.py` 第 66 行)，其建议全部由大模型依赖通用参数盲态生成，而不是结合了爬虫抓取本地数据集动态分布推算的物理决策边界。

## 🎯 4. 异常处理与数据稀疏 (Error Handling)
**审查目标：** 对极端边缘情况（低样本、计算超时）的处理逻辑

### 4.1 数据极窄条件下的平滑与降级逻辑
1. **源码位置：** `d:\renhangxi_tiktok_bysj\ml_pipeline\theme_baseline_engine.py` (第 111-147 行) 。
2. **处理平滑方案 (Smoothing)：** 有严密的**“贝叶斯平滑 (Bayesian Smoothing)”**公式抵御偏差。
   - 如果赛道爬虫数据足够且有全局基准 (Global)，将运用贝叶斯修正常数($K=10$): 
     $$X_{smoothed} = \frac{N \times Local_{statistic} + K \times Global_{statistic}}{N + K}$$
3. **报错降级 (Degradation)：** 具备最低样本守卫。
   - `MIN_SAMPLE_SIZE = 5` (第19行)。当爬虫数据绝收（$N < 5$）同时未能获得 `global_stats` 支持时，代码将启动硬编码全局假设（`DEFAULT_BASELINE`字典, 均值锁定 10000 级, 警告标识）强行顶替。
4. **LLM 预测降级兜底：** `views.py` `predict_api` 存在并发调用池，限定最大并发 1，设置了显微级短效熔断：`timeout=2.0` 秒。一旦响应超时（包括调用 API 或提取 Whisper），会自动向控制台输出并发超时报警并强制降级为一个硬编码的字符串建议返回前台，系统未崩溃。
