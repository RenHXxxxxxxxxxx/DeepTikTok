# Stage 4 Inference Layer Audit Report

## 1. The "Eternal Cache" Vulnerability (Hot-Reloading Failure)
**Issue Assessment:**
The `DiggPredictionService` is implemented as a strict Singleton using the `__new__` and `__init__` methods. Within `__init__`, the service sets `self._initialized = True` immediately after `_load_assets()` executes for the first time. Because of this lock, the service will **NEVER** hot-reload new models generated by `train_model_arena.py`. Models are locked in an "eternal cache", and new weights will not take effect unless the entire Django server process is restarted.

**Proposed Solution:**
Store the last modified timestamp of the manifest using `os.path.getmtime(manifest_path)` as an instance attribute during initialization. On every `predict_digg_count` request, perform a lightweight check to see if the manifest file's current `mtime` has changed. If it has updated since the last check, trigger an on-the-fly `_load_assets()` to seamlessly load the hot new model before issuing the prediction.

## 2. Fake Fallback vs. True Silent Rollback
**Issue Assessment:**
In the `predict_digg_count()` method, the `self.model.predict()` execution is wrapped in a `try-except` block. When a core failure occurs (e.g., Numpy crash or incompatible tensor shape), it safely catches the error but defaults to returning `{"predicted_digg": 0, "quality_score": 0.0}`. This is merely a "Fake Fallback"—it prevents a 500 Server Error but degrades the user experience down to absolute zero.

**Proposed Solution (True Silent Rollback):**
When an exception occurs during `predict`, the `except` block should orchestrate a "True Silent Rollback":
1. Catch the exception and suppress immediate failure.
2. Read the `version_manifest.json` file on disk.
3. Identify and load the `previous_version` (the older, stable `.pkl` model and scaler) into memory.
4. Retry the exact same inference calculation using the fallback model.
5. Surface the successful prediction implicitly to the user without any downtime or zero-score degradation.

## 3. Manifest Version Awareness
**Issue Assessment:**
The `_load_assets()` method is currently blind to the system's deployment history. It explicitly reads the `active_version` metadata and blindly loads those files, completely ignoring the `previous_version` capability established in Stage 3. Because it lacks awareness of the rollback properties of the new `version_manifest.json`, the service is architecturally incapable of executing recovery paths.

**Recommendation:**
Update `_load_assets()` to read and cache the `previous_version`, `previous_model_file`, and `previous_scaler_file` fields from the manifest and store them as instance attributes (e.g., `self.fallback_model`). This primes the service for immediate availability of rollback artifacts.
